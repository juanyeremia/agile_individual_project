AI Reflection Log

Introduction

Throughout this project I used Claude (by Anthropic) as an AI assistant to help guide my learning and implementation of HTML, CSS, and JavaScript. This page documents how I used AI it, what it produced, the corrections I made, and a critical evaluation of its limitations.

Use Case 1: Overall Theming System

Prompt

I wanted to have a consistent look and across all four pages of the project. I consulted with Claude with the initial concept: "I want to have consistent UI elements across the entire webpage. What would be the best way to do this? I was thinking like a header at the top, a navbar that has position sticky so it will stop at the top and follow the scroll, and the content below it as a card." I had a barebones idea in mind and wanted guidance on how to structure it properly.

What Claude Produced

Claude agreed with the general concept and extended it by suggesting a color theming system to keep UI elements visually consistent. It proposed a Forest Terminal color palette, with deep green, emerald, and pale green tones, and introduced me the use of CSS custom properties to implement it. Specifically, it suggested defining all colors once in a :root block using CSS variables like --forest-nav, --forest-accent, and --forest-accent-lt, then referencing them throughout the CSS using var().

What I Changed

Claude initially suggested defining the :root variables inside tutorial.css. I immediately thought that it suggested to do this for each page's CSS. I thought that this would limit the variables to one specific page only, making them inaccessible to the other three pages. I decided it would be more efficient to have a general CSS file, general_structure.css, a shared stylesheet that will loaded by all four pages. This way the color system is defined once and available everywhere, while each page's CSS file only contains styles unique to that page. This decision reflects the separation of concerns principle covered in the CSS lecture.

Use Case 2: Interactive Demo

Prompt

After building the tutorial content, I wanted to add an interactive element to the page as it was required by the rubric. I had no prior experience building interactive components from scratch, so I asked Claude how to create a live HTML editor where users could type HTML and see it rendered in real time.

What Claude Produced

Claude broke down the component into two parts, the HTML structure and the JavaScript logic. For the HTML, it suggested to create two containers side by side, using flexbox with one side being the input area and the right side being the output. It suggested usign <textarea> for user input, a Run button, and an output <div> to display the result. For the JavaScript, it explained how i can display the input as HTML by putting addEventListener on the button that listens for a click event, retrieves the textarea's current value, and writes it into the output div using innerHTML, causing the browser to parse and render the typed HTML live. Claude also explained why each element was needed and how they connected, rather than just providing the code.

What I Changed

Claude initially suggested using an <iframe> to sandbox the output, which would have required accessing contentDocument, which was a method not covered in the lectures. I raised this concern and Claude offered a simpler alternative using innerHTML on a plain <div>, which achieved the same visual result using only DOM manipulation techniques from the JavaScript lectures. Claude also initially wrote the implementation in vanilla JavaScript using document.getElementById and innerHTML. I independently converted this to jQuery by replacing document.getElementById('demo-input') with $('#demo-input') and output.innerHTML with output.html().

Use Case 3: Debugging

Prompt

Throughout the project I used Claude as a debugging companion. My typical workflow was to implement code, test it in the browser, and when something didn't work as expected I would share a screenshot or paste the relevant code and describe the problem. Claude would identify the issue and suggest a fix, which I would then implement and verify before continuing.

What Claude Produced

Claude identified a range of bugs across the project. For example, there was a typo where a CSS class was written as into_par instead of intro_par in the HTML, causing the justify style to not apply. In the quiz page, Claude identified that HTML entities like <nav> and <div> used as answer options were being parsed as actual HTML tags rather than displayed as text, causing them to dissapear. It suggested writing an escapeHTML() function to make the output appear as text before injecting it into the DOM. Claude also caught a missing dot in a jQuery selector where $('question-card') should have been $('.question-card').

What I Changed

In most cases I implemented Claude's suggested fixes directly after understanding the cause of the bug. However, not all bugs were caught by Claude initially. Several bugs were only identified after I asked it about unexpected behaviour in the browser that i could not figure out by myself. This highlights an important limitation, that Claude could not run or test the code itself, meaning I was responsible for executing, observing, and reporting issues. The debugging process was collaborative. Claude provided the diagnosis, but I was the one running the tests and confirming whether the fix worked.

Critical Evaluation

AI as a Collaborator

Throughout this project, Claude functioned most effectively as a collaborator rather than a replacement for my own thinking. The most productive interactions were not when I asked Claude to write something for me, but when I came with an idea or a problem and used Claude's response as a starting point. Final decisions on structure, naming conventions, which approach to use, and what to include were always made by me.

Where AI Added Genuine Value

Claude was particularly useful for identifying issues I would not have caught on my own. The escapeHTML fix for quiz answer options containing HTML tags, the event delegation pattern for dynamically created elements, and the CSS variable theming system were all suggestions that improved the quality of the project beyond what I would have produced myself. In these cases, AI did not just save time, it introduced approaches and patterns I was not yet aware of.

Risks and Limitations

The most significant risk of AI-assisted development i think is the temptation to accept output without understanding it. On several occasions Claude suggested approaches that went beyond the lecture material, such as the iframe sandbox for the interactive demo, a specific shuffle algorithm for the quiz, and try/catch for localStorage. Accepting these without question would have produced working code I would not be able to explain. Instead, I questioned each suggestion, asked Claude to explain the reasoning, and in some cases chose simpler alternatives that I could justify.
